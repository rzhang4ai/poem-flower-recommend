## 第一部分：可视化研究体系

### 1.1 数据预处理：构建意象语义维度矩阵

#### 原理

每首诗的 `sxhy_catalog_paths` 字段记录了该诗命中的所有意象词的完整分类路径。例如：

```
卷一/天文类/月|卷四/百花类/花|卷二/宫室类/观|卷一/时令类/早春|卷二/丽人类/闺感
```

这意味着这首诗涉及了天文（月）、植物（花）、建筑空间（观）、时令（早春）、人物情感（闺感）五个语义范畴。将 40 个小类映射为更高层的语义维度，就能为每首诗构建一个"意象语义指纹"。

#### 映射方案

将 40 个诗学含英小类映射为 8 个高层语义维度：

| 高层维度 | 英文标识 | 诗学含英小类 | 维度含义 |
|---------|---------|------------|---------|
| 自然/天象 | `dim_nature` | 天文类 | 日月星辰、风雨雷电等自然天象 |
| 时令/节序 | `dim_season` | 时令类, 节序类 | 四季物候、传统节日 |
| 地理/空间 | `dim_space` | 地舆类, 宫室类, 游眺类 | 山川地貌、建筑场所、游览胜景 |
| 人物/关系 | `dim_people` | 人伦类, 师友类, 丽人类, 人品类, 君道类, 臣道类, 百官类 | 人际关系、社会角色、品貌评价 |
| 品格/志向 | `dim_virtue` | 志气类, 仕进类, 仕宦类, 释老类 | 人格品质、仕途抱负、宗教哲学 |
| 器物/生活 | `dim_artifact` | 器用类上, 器用类下, 服饰类, 珍宝类, 饮食类, 音乐类, 书画类 | 日用器物、饮食衣着、文艺活动 |
| 动植物/生态 | `dim_biota` | 百花类, 树木类, 竹木类, 百草类, 飞禽类, 走兽类, 昆虫类, 鳞介类, 百果类, 百谷类, 蔬菜类 | 花草树木、鸟兽鱼虫 |
| 人文/社交 | `dim_social` | 人事类, 文学类, 庆吊类, 祖饯类, 谢惠类 | 人生际遇、文学创作、社交礼仪 |

#### 为什么这样分？

- **自然/天象** 独立成维度，因为它在古诗中既是写景工具，又是最核心的"兴"（起兴）手法的载体
- **时令/节序** 独立，因为它直接关联"月份"这一推荐维度，且是花卉咏物诗的基本时间锚点
- **品格/志向** 与 **人物/关系** 分开，因为咏花诗中"品格"多指花的象征（如梅花=高洁），而"人物"指诗中涉及的人际关系
- **动植物/生态** 内含"百花类"——花卉诗中花自身及伴生植物/动物属于核心主题

#### 同时：BERT token 与诗学含英的交叉确认

将 `ccpoem_top10` 中的 token 与 `sxhy_raw_words` 做交集：
- 交集中的词 = **双重确认意象**：BERT 注意力机制认为重要 + 传统词典收录
- 这些词的可信度最高，是后续推荐系统做关键词匹配的首选

#### 输出

在 `poems_structured_merged.csv` 基础上扩展为新文件，新增列：
- `dim_nature`, `dim_season`, `dim_space`, `dim_people`, `dim_virtue`, `dim_artifact`, `dim_biota`, `dim_social`（8 个整数，每个为该维度命中的意象词条数）
- `dim_vector`（8 维向量的归一化形式，便于直接计算距离）
- `confirmed_imagery`（ccpoem_top10 ∩ sxhy_raw_words 的交集词列表）

### 1.2 可视化研究课题

#### 课题 A：花卉意象语义画像（雷达图）

**研究问题**：不同花在意象语义空间中的"形状"有何差异？

**原理**：每种花聚合其所有诗的 8 维语义向量（取均值后归一化），就得到该花的"语义指纹"。不同花的雷达图形状差异，直接反映了古人在咏该花时偏好的意象范畴。

**预期发现**（基于对数据的初步理解）：
- 梅花：在"自然/天象"(霜/雪/风)和"品格/志向"维度突出——对应其"傲寒高洁"的文化象征
- 牡丹：在"器物/生活"和"人物/关系"维度突出——对应其"国色天香""富贵"的宫廷/社交意象
- 菊花：在"品格/志向"和"时令/节序"维度突出——对应"东篱归隐"和重阳节的文化联结
- 荷花/莲花：在"动植物/生态"和"品格/志向"维度突出——对应"出淤泥而不染"的君子象征

**可视化方案**：
- 选取诗数 >= 20 的花（约 10-12 种），生成并排雷达图（2×6 子图矩阵）
- 可用 plotly 做交互版，hover 显示具体意象词
- 额外：所有花的 8 维向量做 PCA 降维到 2D，生成花卉"语义地图"

**学术价值**：这类系统性的多花横向对比在现有研究中罕见。Gong & Zhou (2025) 仅对比了牡丹/梅花 2 种，而我们覆盖 79 种。

#### 课题 B：花卉-朝代意象演变（热力图 + 面积流图）

**研究问题**：同一种花的意象内涵在不同朝代如何漂移？

**原理**：将花的诗按朝代分组，分别计算各组的 8 维语义向量。不同朝代间向量的变化，量化揭示了该花"文化含义的历时演变"。

**朝代排序与合并策略**：
数据中 15 个朝代样本极不均衡（宋 518、唐 215，而多个南朝/北朝朝代仅 1-10 首）。建议合并为 6-7 个时段：

| 合并时段 | 包含朝代 | 诗数 |
|---------|---------|-----|
| 魏晋南北朝 | 魏, 晋, 南朝宋/齐/梁/陈, 北朝周 | ~23 |
| 隋唐 | 隋, 唐 | ~219 |
| 五代 | 五代 | 2 |
| 宋金 | 宋, 金 | ~549 |
| 元 | 元 | 58 |
| 明 | 明 | 100 |
| 清 | 清 | 124 |

因"五代"仅 2 首，可并入"宋金"或忽略。最终得到 6 个有效时段。

**重点分析花卉**（跨朝代样本充足）：
- 梅花（161首）：宋101, 唐18, 清13, 元10, 明7
- 菊花（109首）：宋38, 唐23, 明24, 清12
- 牡丹（53首）：唐24, 宋21
- 桃花（45首）：宋21, 唐11

**可视化方案**：
1. **热力图**：行=合并时段，列=8维语义维度，色=占比强度。每种花一张，4 张并排对比
2. **堆叠面积图**（alluvial）：横轴=时段，纵轴=各维度占比堆叠，展示维度消长的"流动感"
3. 可叠加情感层：在热力图上用第二色阶（如冷暖色）表示该时段该花的 positive/negative 比例

**预期洞见**：
- 梅花：唐代偏"自然/天象"（实写凌寒/报春），宋代"品格/志向"维度显著抬升（受理学和文人精神影响）
- 菊花：晋-唐以"品格/志向"为主（陶渊明影响），明代"器物/生活"维度升高（菊花栽培文化兴盛）
- 牡丹：唐代"人文/社交"极高（宫廷赏花/国花地位），宋代回落

#### 课题 C：花卉情感全景图（桑基图 + 堆叠柱状图）

**研究问题**：不同花携带的情感分布有何系统性差异？同一情感在不同花中的意象表达有何不同？

**原理**：情感分类器提供了三层情感标签（L1→L2→L3）。桑基图（Sankey Diagram）天然适合展示这种层级流向关系，从花名出发，经 L2 粗分类到 L3 细分类，每条"流"的宽度代表诗数。

**可视化方案**：
1. **桑基图**：左列=花名（Top10），中列=L2（positive/neg_sorrow/neg_anger），右列=L3（10类）
   - 流宽=诗数
   - 例如：梅花 → positive(53) → praise(39)/like(4)/joy(8)/faith(2)
   - 梅花 → neg_sorrow(33) → sorrow(22)/miss(11)
   - 梅花 → neg_anger(38) → criticize(20)/vexed(17)/misgive(1)
   
2. **100%堆叠柱状图**：每种花一根柱，内部按 L3 占比着色
   - 按主导情感排序：praise 占比从高到低
   - 颜色方案：暖色系=positive 类（praise/like/joy/faith/ease），冷色系=negative 类（sorrow/miss/criticize/vexed/misgive）

3. **情感-意象交叉分析**（深度分析）：
   - 对于同一 L3 情感（如 sorrow），提取不同花中该情感诗的意象词频
   - 例如：梅花的"sorrow"诗中高频意象 = {寒/夜/月/断/独}，菊花的"sorrow"诗中 = {秋/残/黄/晚/霜}
   - 这揭示了"相同情感通过不同意象系统表达"的规律

#### 课题 D：BERT token × 诗学含英交叉分析（韦恩图 + 词云）

**研究问题**：深度学习模型（BERT 注意力权重）与传统诗学词典识别的"重要词"有多大重叠？差异部分说明了什么？

**原理**：
- **BERT-CCPoem** 的注意力权重反映的是"模型在大规模古诗语料上学到的上下文重要性"——这是数据驱动的、语境敏感的
- **诗学含英词典** 反映的是"清代学者基于诗学传统手工编纂的诗歌用语分类"——这是知识驱动的、文化编码的
- 两者的交集 = 模型和传统知识一致认可的核心意象，可信度最高
- BERT 独有 = 模型发现的隐性重要词（可能是虚词、连接词、或新型意象组合）
- 词典独有 = 传统认可但在当前诗歌上下文中 BERT 认为不突出的程式化用语

**可视化方案**：
1. **韦恩图**（全局）：所有诗的 ccpoem_top10 tokens 集合 vs sxhy_raw_words 集合，展示全局重叠率
2. **按花分类韦恩图**：对 Top10 花各做一张，对比不同花的"双重确认率"
3. **着色词云**：对每种花，将所有出现的意象词按来源着色：
   - 红色 = 双重确认（交集）
   - 蓝色 = BERT 独有
   - 绿色 = 词典独有
   - 字号 = 频次/权重

**学术价值**：这是一个"可解释 AI"视角的分析——用传统知识体系验证深度学习模型的注意力分配，或反过来用模型发现传统词典的盲区。

#### 课题 E：诗人-花-意象偏好网络（交互式网络图）

**研究问题**：高产诗人对特定花卉的意象运用是否有个人风格？

**原理**：每位诗人在咏同一种花时，选取的意象词和语义维度可能不同，这反映了诗人的个性化美学偏好。用三层网络图（诗人 → 花 → 意象维度）可以直观展示这种关联。

**筛选策略**：
- 仅选作品 >= 5 首的诗人（预计 20-30 人）
- 仅选诗数 >= 10 的花（预计 15-20 种）
- 边权 = 诗数或语义维度均值

**可视化方案**：
- 用 pyvis 或 networkx + D3.js 生成交互式 HTML
- 节点大小 = 作品数 / 诗数 / 意象频次
- 节点颜色 = 类型（诗人/花/意象维度）
- 边粗细 = 关联强度
- 交互：点击诗人节点展示其全部花-意象关联

#### 课题 F：花卉月令图谱（极坐标/环形图）

**研究问题**：79 种花在 12 个月的分布如何？相邻月份的花之间有何语义关联？

**原理**：原书按月份编排花卉，每种花有其"对应月份"。极坐标图以 12 个月为角度轴，天然适合展示这种环形时间结构。叠加情感和意象信息后，可以揭示"时间-花-情感"的三维关系。

**可视化方案**：
1. **极坐标散点图**：
   - 角度 = 月份（1-12月均匀分布在 360°）
   - 半径 = 该月该花的诗数
   - 颜色 = 主导情感（positive 系=暖色，negative 系=冷色）
   - 每个点旁标注花名

2. **月份-花热力矩阵**：
   - 行=12 月，列=主要花卉（按首次出现月份排序）
   - 色=诗数
   - 用于展示花期分布的"时间窗口"

3. **花期邻近网络**：
   - 将月份相邻（±1月）且意象维度相似（余弦>阈值）的花连线
   - 这直接服务于推荐系统的"3 月窗口"逻辑——如果用户 3 月送花，除了 3 月的花，哪些 2 月/4 月的花在意象上与用户需求最接近？

**与推荐系统的直接关联**：此图谱是推荐系统"月份过滤层"的可视化验证和解释工具。

## 第二部分：可追溯诗歌与花推荐系统

### 2.1 设计理念

**核心区别于 LLM 推荐**：本系统的每一步匹配都基于结构化数据和明确的计算规则，用户可以追溯"为什么推荐这首诗/这种花"。LLM 推荐是黑盒——它可能给出合理的结果，但无法解释是根据哪条诗、哪个意象、哪个情感维度做出的判断。

**可追溯性**体现在：
1. 月份匹配可追溯：明确显示"花期在哪几个月，用户指定月份是否在窗口内"
2. 情感匹配可追溯：明确显示"用户期望的情感 vs 诗歌被分类器判定的情感 + 概率值"
3. 意象匹配可追溯：明确显示"用户输入的关键词在诗中对应哪些意象词、属于哪个诗学含英分类"
4. 语义维度匹配可追溯：明确显示"用户需求映射到哪些语义维度、诗歌在这些维度的得分"

### 2.2 系统架构

```
用户自由输入（文本 / 结构化选项）
        │
        ▼
┌───────────────────────────────────┐
│  层1：意图解析                      │
│  · 提取月份 → 月份数字              │
│  · 提取情感关键词 → 映射到 L1/L2/L3  │
│  · 提取场景/对象关键词               │
```

### 2.3 各层详细设计

#### 层1：意图解析

用户可能输入自由文本，需要提取结构化意图。不依赖大模型，用规则 + 关键词词典实现：

| 意图维度 | 提取方法 | 示例 |
|---------|---------|------|
| 月份 | 数字/中文月份正则 + "春/夏/秋/冬"季节映射 | "三月"→3, "春天"→{2,3,4} |
| 情感 | 情感关键词词典（敬意→praise, 思念→miss, 哀悼→sorrow, ...） | "表达敬意"→L3=praise |
| 对象关系 | 关系词词典（老师/朋友/恋人/长辈/自己/...） | "送老师"→teacher |
| 场景关键词 | 直接传递给 BM25 | "高洁""思念故乡" |

**情感映射词典（核心资源，需人工构建）**：

| 用户可能的表达 | 映射到 L3 | 映射到 L1 概率向量 |
|--------------|----------|-----------------|
| 敬意、敬佩、尊敬 | praise | [0, 0, 0.1, 0.3, 0.6] |
| 思念、想念、怀念 | miss | [0.1, 0.5, 0.1, 0.3, 0] |
| 安慰、宽慰 | ease | [0, 0.2, 0.2, 0.5, 0.1] |
| 喜悦、祝贺、开心 | joy | [0, 0, 0, 0.2, 0.8] |
| 哀悼、悼念、伤逝 | sorrow | [0.6, 0.3, 0.1, 0, 0] |
| 激励、鼓励、勉励 | faith | [0, 0, 0.1, 0.4, 0.5] |
| 讽刺、批评 | criticize | [0.3, 0.5, 0.1, 0.1, 0] |

#### 层2：月份过滤（3月窗口）

**原理**：原书按传统花期编排（正月梅花、二月杏花...），但当代花卉栽培技术（温室/催花）可将花期前后调节约 1 个月。因此用 3 个月的窗口而非严格匹配。

**实现**：

```python
def month_filter(df, target_month: int | None) -> pd.DataFrame:
    """
    target_month: 1-12 的整数，None 表示不过滤
    返回月份在 [target_month-1, target_month, target_month+1] 窗口内的诗
    """
    if target_month is None:
        return df
    window = {(target_month - 2) % 12 + 1,
              target_month,
              target_month % 12 + 1}
    return df[df['月份数字'].isin(window)]
```

示例：用户指定"三月" → 窗口={2,3,4} → 候选诗从1075首缩减到约438首（二月189+三月180+四月69）。

#### 层3：多维匹配

**3a. 意象关键词匹配（BM25）**

原理：BM25 是经典的文本相似度算法，在信息检索领域广泛使用。它考虑了词频、文档长度和逆文档频率，比简单的关键词计数更准确。

对每首诗，将 `sxhy_raw_words` 和 `ccpoem_top10` 合并为一个"意象文档"，用户输入作为查询，计算 BM25 分数。

**3b. 情感维度匹配（余弦相似度）**

原理：每首诗有 5 维 L1 概率向量 `[prob_negative, prob_implicit_negative, prob_neutral, prob_implicit_positive, prob_positive]`。用户的期望情感也映射为同样的 5 维向量。两者的余弦相似度衡量情感方向的一致程度。

余弦相似度的优势：只关心"方向"不关心"大小"，即使模型对某首诗的概率分布较平坦（不确定），只要方向与用户期望一致，仍能得到较高的匹配分。

**3c. 语义维度匹配（8维向量距离）**

原理：将用户输入映射到 8 维语义空间后，与每首诗的 `dim_vector` 计算余弦相似度。

用户输入如何映射到 8 维？两种策略：
1. **关键词匹配**：将用户输入分词后查诗学含英词典，命中的词按所属类别累加到对应维度
2. **语义模板**：预定义一些场景模板（如"送别"→ dim_social 高, dim_nature 中, dim_people 中），根据意图解析结果选择模板

**3d. BERT 语义匹配（可选）**

原理：`ccpoem_cls_l2` 存储了每首诗通过 BERT-CCPoem 编码后的 [CLS] 向量的 L2 范数。若加载完整的 CLS 向量（需从 step2b 输出的 npy 文件获取），可以将用户输入也编码为 768 维向量，做更细粒度的语义匹配。

这一步是可选的，因为需要加载 BERT 模型（约 400MB），增加部署复杂度。在轻量化部署时可省略。

#### 层4：融合排序（RRF）

**原理**：Reciprocal Rank Fusion 是一种无参数（仅一个常数 k）的排名融合方法。它不依赖分数的绝对值（不同维度的分数量纲不同），只依赖排名，因此天然适合融合异构维度。

公式：

$$\text{RRF\_score}(d) = \sum_{i=1}^{n} \frac{w_i}{k + \text{rank}_i(d)}$$

其中 $d$ 是候选诗，$\text{rank}_i(d)$ 是该诗在第 $i$ 个维度上的排名，$w_i$ 是该维度的权重，$k=60$ 是标准平滑常数。

默认权重（可根据用户输入动态调整）：
- 意象关键词匹配 $w_1 = 0.3$
- 情感维度匹配 $w_2 = 0.3$
- 语义维度匹配 $w_3 = 0.2$
- 月份匹配度 $w_4 = 0.2$

**动态权重调整**：
- 用户输入包含明确情感词（"敬意""悲伤"）→ 情感权重上调至 0.4
- 用户输入包含具体意象词（"月""雪""梅"）→ 意象权重上调至 0.4
- 用户未指定月份 → 月份权重置 0，其余均分

### 4.2 核心结论：Streamlit 不可行，需要独立前端

**Streamlit 的根本局限**：

Streamlit 的设计哲学是"Python 脚本驱动的数据面板"，其架构有以下硬伤无法满足本构想：

1. **iframe 沙箱隔离**：自定义组件跑在 iframe 里，组件之间无法通信，无法实现"角色动画 + 聊天窗口 + 背景图 + 诗词展示"的联动
2. **全页面重渲染**：每次用户交互都会触发整个 Python 脚本重新执行，角色动画会被打断/闪烁
3. **无持久状态**：Canvas/WebGL 动画需要帧级持久状态（角色位置、表情、动画帧），Streamlit 的 session_state 无法支撑 60fps 动画循环
4. **无 WebSocket 双向推送**：ComfyUI 图片生成耗时 10-30 秒，需要 WebSocket 实时推送进度，Streamlit 不支持自定义 WebSocket

虽然 Streamlit Custom Components v2（2025 年底预览版）移除了 iframe 隔离，但仍然不适合构建游戏级交互——它解决的是数据组件通信，而非实时动画渲染。

### 4.3 各模块可行性与难度评估

#### M1 花朵角色系统 — 可行，中等难度

**技术方案**：Pixi.js + Spine 骨骼动画

- 每种花设计一个 Spine 角色（骨骼 + 皮肤），包含以下动画状态：
  - `idle_bounce`：闲置弹跳（进入页面时）
  - `wake_up`：被选中唤醒
  - `emotion_happy` / `emotion_sad` / `emotion_angry` / `emotion_thinking`：表情切换
  - `grow_stem`：生长花梗和四肢
  - `walk` / `sit` / `climb`：在场景中移动的动作

- **村上隆太阳花风格**：需要美术设计师绘制每种花的原画，然后在 Spine Editor 中绑定骨骼。79 种花全部设计不现实，建议先做 12 种（每月代表花），后续逐步扩展。

| 子项 | 工作量 | 技能需求 |
|------|--------|---------|
| 花朵原画设计（12种 × 村上隆风格） | 5-8 天 | 插画师/美术 |
| Spine 骨骼绑定 + 动画制作（12种 × 6 个动画状态） | 8-12 天 | Spine 动画师 |
| Pixi.js + Spine Runtime 前端集成 | 3-5 天 | 前端开发 |
| 表情状态机（情绪 → 动画映射） | 2 天 | 前端开发 |

**风险**：美术资源是最大瓶颈。Spine 动画质量直接决定体验。如果没有专业 Spine 动画师，可退化为帧动画（sprite sheet），但表现力会大幅下降。

#### M2 对话系统 — 可行，低难度

**技术方案**：前端 Chat UI + FastAPI + LLM API

- 前端用现成的 React Chat 组件（如 chatscope/chat-ui-kit-react）
- 后端用 FastAPI 接收消息，调用 LLM API（OpenAI / DeepSeek / 本地 Qwen）
- LLM prompt 中注入花朵角色人设（"你是桃花，性格温婉..."）

| 子项 | 工作量 | 技能需求 |
|------|--------|---------|
| Chat UI 前端 | 2 天 | 前端开发 |
| FastAPI 对话后端 | 2 天 | Python 后端 |
| LLM prompt 设计（12种花 × 人设） | 2 天 | 提示工程 |

**风险**：低。成熟技术，无难点。

#### M3 情绪识别与映射 — 可行，低-中难度

**技术方案**：LLM 结构化输出 + 前端动画状态机

- LLM 返回 JSON：`{"emotion": "sorrow", "intensity": 0.7, "keywords": ["想念", "故乡"]}`
- 可直接复用我们已有的情感分类器（L1/L2/L3）对 LLM 输出做二次验证
- 前端收到情绪后触发 Spine 动画切换
- 无情绪时：展示 3-4 个不同表情的花朵小图标，用户点选

| 子项 | 工作量 | 技能需求 |
|------|--------|---------|
| LLM 情绪提取 prompt + 结构化输出 | 1 天 | 提示工程 |
| 前端情绪→动画联动 | 2 天 | 前端开发 |
| 无情绪时的多表情选择 UI | 1 天 | 前端开发 |

**风险**：低。LLM 情绪识别准确率对现代语言足够高。

#### M4 意象匹配与图像生成 — 可行，中-高难度

**技术方案**：后端意象匹配 + ComfyUI API + 吴冠中风格 LoRA

- **意象匹配**：复用 `05.recommend/` 的意象关键词匹配逻辑，从用户对话中提取场景/物品/自然等意象词，查诗学含英词典的分类路径
- **prompt 构建**：将匹配到的意象词填入预设的 prompt 模板，如：
  ```
  "Wu Guanzhong style ink painting, minimalist brushstrokes, low saturation colors,
   [scene: 月下亭台], [elements: 寒梅, 石阶, 远山], [mood: 清冷孤寂],
   simple line art, Chinese landscape, white space"
  ```
- **ComfyUI 工作流**：预设工作流模板，通过 API 动态替换 prompt 中的意象部分
- **吴冠中风格**：需要训练或寻找社区 LoRA。Civitai 上有部分中国水墨风格 LoRA，但精确的"吴冠中简化线条"风格可能需要自训

| 子项 | 工作量 | 技能需求 |
|------|--------|---------|
| 意象词 → prompt 模板系统 | 3 天 | Python + 提示工程 |
| ComfyUI 工作流设计 + API 集成 | 3-5 天 | ComfyUI + Python |
| 吴冠中风格 LoRA 训练/调试 | 5-10 天 | AIGC / LoRA 训练 |
| 前端背景图动态替换 | 2 天 | 前端开发 |

**风险**：
- LoRA 风格一致性：生成图的风格可能不稳定，需要大量 prompt 调试
- 生成速度：SDXL 单张图 10-30 秒（取决于 GPU），用户等待体验需设计loading动画
- GPU 成本：需要至少一张 A10/A100 级显卡运行 ComfyUI（云 GPU 约 $0.5-1/小时）

#### M5 场景交互（花朵进入生成画面）— 部分可行，高难度

**这是整个构想中技术难度最高的模块。**

用户的设想是：ComfyUI 生成一幅吴冠中风格的意境图后，花朵角色要"走进"这幅画，识别画中可交互的位置（台阶、石头、桥等），然后做出对应动作（走到台阶上、坐在石头上）。

**难点分析**：

1. **场景语义理解**：需要理解生成图中哪些区域是"可站立的地面"、"可坐的石头"、"可攀爬的台阶"——这本质上是**语义分割 + 物理可达性推理**
2. **2D 角色与静态画面的融合**：角色是 Spine 动画（矢量/骨骼），背景是 AI 生成的位图，两者的视觉风格需要协调
3. **遮挡关系**：角色走到"石头后面"需要正确的前后遮挡

**可行方案（三档，复杂度递增）**：

**方案一（简单，推荐初期采用）：预定义交互区 + 受限构图**

- 不做通用场景理解，而是**约束 ComfyUI 生成图的构图**
- 预设 5-8 种构图模板（如"前景有石头的山水"、"有台阶的庭院"、"有桥的溪边"）
- 每种模板预定义好交互热区坐标（石头在画面 60%-70% 高度处，台阶在底部 80% 处等）
- 花朵角色只需要移动到预定义坐标并播放对应动画

| 子项 | 工作量 |
|------|--------|
| 构图模板设计（5-8种） | 3 天 |
| ComfyUI ControlNet 约束构图 | 3-5 天 |
| 前端预定义热区 + 角色移动 | 3 天 |
| **小计** | **9-13 天** |

**方案二（中等）：SAM 3 自动分割 + LLM 场景描述**

- ComfyUI 生成图后，用 SAM 3（Segment Anything Model 3）做全图语义分割
- 分割结果送 GPT-4o / Qwen-VL 多模态 LLM，让它描述："图中有一块石头在坐标 (340, 520) 附近，适合角色坐下；有台阶从 (100, 600) 延伸到 (300, 450)"
- 前端根据 LLM 返回的坐标和类型，动态生成交互热区

| 子项 | 工作量 |
|------|--------|
| SAM 3 服务部署 + API | 3 天 |
| 多模态 LLM 场景描述 prompt | 3 天 |
| 后端场景理解管线 | 5 天 |
| 前端动态热区 + 角色动画 | 5 天 |
| **小计** | **16-20 天** |

**方案三（复杂）：深度估计 + 2.5D 场景**

- 用 Depth Anything V2 / MiDaS 对生成图做深度估计，得到像素级深度图
- 将 2D 画面转化为伪 3D 场景（近景/中景/远景层）
- 花朵角色在深度层之间移动，自动处理遮挡和缩放
- 用 SAM 3 分割出"可交互对象"，结合深度确定 3D 位置

| 子项 | 工作量 |
|------|--------|
| 深度估计 + 层分离 | 5 天 |
| 2.5D 场景渲染（Pixi.js 多层） | 8 天 |
| SAM 3 + 深度融合 | 5 天 |
| 角色在 2.5D 场景中的物理约束 | 8 天 |
| **小计** | **26-30 天** |

**建议**：初期采用方案一，验证核心体验后再升级到方案二。方案三是锦上添花，性价比不高。

#### M6 诗词推荐展示 — 可行，中等难度

#### M7 扩展模块（插花/绘画） — 规划阶段，暂不评估

### 4.5 风格统一性分析

用户提出的"村上隆太阳花风格角色 + 吴冠中风格背景"在视觉上是一个大胆的混搭：

- **村上隆**：高饱和度、平面化、波普艺术、笑脸花朵、强烈的装饰感
- **吴冠中**：低饱和度、写意线条、水墨晕染、留白、意境感

两者反差极大，但用户说"形成一种穿越感"——这在概念上是成立的：
- 背景（吴冠中）= 古典诗意世界
- 角色（村上隆）= 当代/超现实的"来访者"
- 穿越感 = 一个当代波普风格的角色走进了一幅水墨画

**技术上的协调方案**：
- 花朵角色加微妙的水墨边缘线（blend 两种风格的过渡）
- 角色投影/阴影与背景色调统一（低饱和度阴影）
- 角色在背景中时适度降低饱和度（通过 shader 混合）

### 4.6 对比：纯 Streamlit 方案能做到什么

如果坚持用 Streamlit，最多能实现的是：

| 功能 | Streamlit 能做到 | 完整构想需要 |
|------|-----------------|-------------|
| 月份花卉选择 | 按钮/下拉菜单 | 弹跳花朵角色点选 |
| 对话 | st.chat_input | 同 |
| 情绪展示 | emoji/静态图切换 | 花朵表情骨骼动画 |
| 意境图生成 | 显示返回图片 | 图片作为交互式背景 |
| 场景交互 | 不可能 | 角色在场景中移动 |
| 诗词展示 | markdown/表格 | 花瓣飘落展开动画 |

**Streamlit 适合做推荐系统的"功能验证原型"，但无法承载游戏化体验。** 建议路线：先用 Streamlit 做 P1-P3 的推荐系统原型 → 验证核心推荐逻辑 → 再启动独立前端项目实现游戏化体验。
